# Outbound email (SMTP)

The app can deliver every email it produces through **your own SMTP
server** — registration/auth emails and team invites alike. SMTP is
optional: with nothing configured, invites remain share-link-only and
Supabase's built-in mailer keeps sending auth emails (fine for
development, but it is heavily rate-limited — a few emails per hour —
and not meant for production).

There are two independent pieces. You can enable either or both.

| What                                       | Emails covered                                                                        | Needs                                                                  |
| ------------------------------------------ | ------------------------------------------------------------------------------------- | ---------------------------------------------------------------------- |
| **A. Auth emails via the Send Email hook** | Signup confirmation, password reset, magic link, email change, reauthentication codes | `SMTP_*` vars + `SEND_EMAIL_HOOK_SECRET` + hook configured in Supabase |
| **B. Team-invite emails**                  | "Invite member" dialog mails the invite link to the new teammate                      | `SMTP_*` vars only                                                     |

## 1. Configure the SMTP relay

Set these in your deployment's environment (see
[.env.local.example](../.env.local.example)):

```bash
SMTP_HOST=smtp.example.com            # required
SMTP_FROM="Tachyel CRM <no-reply@example.com>"  # required
SMTP_USER=no-reply@example.com        # usually required by the relay
SMTP_PASS=your-smtp-password
SMTP_PORT=587                         # default 587 (STARTTLS)
SMTP_SECURE=false                     # "true" for implicit TLS (port 465)
```

Any standards-compliant relay works: Amazon SES, Postmark, Brevo,
Mailgun, Gmail/Google Workspace (app password), or a self-hosted
Postfix. Make sure the `SMTP_FROM` domain has SPF/DKIM set up with
your provider or your mail lands in spam.

With just these vars set, **part B works immediately**: the
"Invite member" dialog (Settings → Members) gains an optional
_Invitee email_ field, and the server mails the one-time invite link.
Sending is best-effort — the link is still shown in the dialog, so a
failed send never strands you.

## 2. Route Supabase auth emails through your SMTP (part A)

Registration and other auth emails are generated by **Supabase Auth**,
not by this app, so Supabase has to be told to hand them to us. That's
what the [Send Email hook](https://supabase.com/docs/guides/auth/auth-hooks/send-email-hook)
does: on every auth email, Supabase POSTs the message data to our
endpoint and we render + send it via your SMTP relay.

1. Deploy the app with the `SMTP_*` vars from step 1. The endpoint is
   `https://<your-domain>/api/auth/send-email` (must be publicly
   reachable over HTTPS).
2. In the Supabase Dashboard go to **Authentication → Hooks**,
   create a **Send Email** hook of type **HTTPS** and point it at that
   URL.
3. Supabase generates a webhook secret (`v1,whsec_...`). Copy it into
   the app's environment as `SEND_EMAIL_HOOK_SECRET`, verbatim, and
   restart/redeploy.
4. Test: sign up with a fresh address (or use "Forgot password"). The
   email should arrive from your `SMTP_FROM` address.

The endpoint verifies the [Standard Webhooks](https://www.standardwebhooks.com/)
signature on every call and rejects anything unsigned, replayed
(±5 min tolerance), or signed with the wrong secret — without the
check, anyone who discovered the URL could pump spam through your
relay.

If the hook is enabled but the app can't send (relay down, bad
credentials), the endpoint returns an error and the triggering auth
action fails visibly — Supabase will not silently fall back to its
own mailer. Disable the hook in the dashboard to revert.

### Alternative: Supabase's built-in custom SMTP

If you don't need custom templates or per-deployment control, you can
skip the hook entirely: **Project Settings → Authentication → SMTP
Settings** in the Supabase dashboard lets Supabase send its own
templates through your SMTP server. In that case leave
`SEND_EMAIL_HOOK_SECRET` unset and don't create the hook — but the
`SMTP_*` vars here are still needed if you want emailed invites
(part B).

## What gets sent

Auth emails rendered by the hook endpoint
(`src/lib/email/templates.ts`):

- **signup** — confirm-your-email with verification link + OTP fallback
- **recovery** — password reset link
- **magiclink** — one-time sign-in link
- **invite** — Supabase-level user invitation (from the Supabase dashboard/admin API; distinct from the CRM's own invite links)
- **email_change** — confirm the new address
- **reauthentication** — 6-digit OTP, no link
- unknown/future types — generic "verify your email" rendering

Team-invite emails (`POST /api/account/invitations` with an `email`
field) carry the account name, assigned role, expiry, and the one-time
`/join/<token>` link.
